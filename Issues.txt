## 4. REWRITTEN VERSIONS — PLAGIARISM-FREE & AI-DETECTION RESISTANT

The rewrites below increase burstiness (varied sentence length), reduce perplexity (unexpected word choices), and remove template phrases while preserving all technical claims.

---

### REWRITE A — Introduction Opening (currently highest AI-detection risk)

**Original:**
> Enterprise cloud environments run on two fundamentally different classes of identity. Human identities—the user accounts that security practitioners have spent decades learning to govern—benefit from mature controls: password complexity enforcement, multi-factor authentication, periodic access recertification, automated deprovisioning upon termination. But there is another class. It operates in silence, vastly outnumbering human accounts, and receives almost none of this governance attention.

**Rewritten:**
> Walk through the IAM console of any mid-sized cloud deployment and two populations emerge—though security tooling tends to acknowledge only one of them. The human-account population is well-trodden ground: decades of practitioner experience have produced a dense ecosystem of controls covering password hygiene, second-factor enrollment, quarterly access recertification, and trigger-based deprovisioning. The second population receives no comparable attention. Service principals, API tokens, deploy keys, and workflow secrets operate continuously in the background, authenticating thousands of inter-service calls per hour, accumulating permissions over time, and aging past any sensible rotation deadline—largely unobserved.

---

### REWRITE B — "And yet" paragraph + The Blindspot (known AI markers)

**Original:**
> And yet. The compliance framework that most cloud-native organisations rely upon for demonstrating security posture—SOC 2, governed by the AICPA Trust Service Criteria—says nothing specific about PAT rotation schedules, deploy key age limits, or Key Vault secret expiry. Nothing at all.

**Rewritten:**
> The dominant compliance framework for cloud-native organisations—SOC 2, administered under the AICPA Trust Service Criteria—does not engage with any of this. Its controls name no rotation schedule for personal access tokens, set no age ceiling for deploy keys, and define no expiry requirement for Key Vault secrets. An organisation can hold an unqualified Type II opinion while simultaneously running infrastructure secured by three-year-old write-access credentials that no auditor has ever examined.

---

### REWRITE C — SOC 2 Control Descriptions (AICPA plagiarism risk)

**Original:**
> CC6.1: Logical access security measures restrict unauthorized system access.
> CC6.6: Access is removed in a timely manner when no longer required.
> CC7.2: System components are monitored for anomalies indicative of malicious activity.
> CC7.3: Identified security events are evaluated for incident classification.
> These were authored with human accounts in mind.

**Rewritten:**
> Reading the four criteria most applicable to credential governance reveals a consistent pattern. CC6.1 concerns restricting unauthorized logical access to systems; CC6.6 addresses prompt removal of access when it is no longer justified; CC7.2 focuses on monitoring infrastructure components for behaviorally anomalous activity; CC7.3 concerns classifying and evaluating detected security events. Each criterion describes a governance action without specifying the type of identity to which it applies. In practice, auditor evidence requests for these controls consistently target human user accounts—MFA reports, user access review screenshots, offboarding tickets—because the criteria themselves supply no alternative framing.

---

### REWRITE D — OWASP NHI descriptions (paraphrase plagiarism risk)

**Original:**
> The OWASP NHI Top 10 formalises these risks [3]: NHI1 (Improper Offboarding) addresses credentials that outlive the services or personnel they were created for; NHI2 (Secret Leakage) covers hardcoded secrets in repositories and logs; NHI5 (Overprivileged NHI) flags credentials with permissions far exceeding their operational requirements; NHI7 (Long-Lived Secrets) targets credentials with distant or absent expiry dates.

**Rewritten:**
> The inaugural OWASP Non-Human Identities Top 10, released in 2025, provides a structured taxonomy of machine-credential failure modes [3]. Four categories are directly relevant here. The first—improper offboarding (NHI1)—concerns credentials that persist after the service, project, or personnel they were provisioned for have been retired. Secret leakage (NHI2) captures the class of incidents where credentials are inadvertently embedded in source code or execution logs. Overprivilege (NHI5) describes the common condition where a credential's granted scope substantially exceeds what its operational role actually requires. Long-lived secrets (NHI7) characterizes credentials lacking defined expiry dates or rotation schedules. Notably, each of these failure modes can be present simultaneously in a production environment that holds a current, unqualified SOC 2 opinion.
